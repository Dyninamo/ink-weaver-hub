## Decisions

### 1. `fishing_style` column → SKIP the session-level write

**Decision: take Lovable's Option 2.** Don't write `fishing_style` on `fishing_sessions`. Style lives on `session_rods.style` only. No schema change.

**Rationale:** The prompt header explicitly says no schema changes; §8a was wrong. `session_rods` is the canonical rod-state row that prompt 142's catch flow reads from. Style is a per-rod concept in a multi-rod future. Aggregating sessions by style requires a join through `session_rods` — acceptable trade-off.

### 2. `fishing_type` capture → MOVE to the precondition block

**Decision: capture `fishing_type` in a "Session basics" block above the wizard, NOT inside SpotStep.**

**Rationale:** Bank / Boat / Both is venue-context, not rig-context. It pairs naturally with venue + date + arrival time. SpotStep stays focused on the rig's spot/plan/keep_limit.

UI: a 3-chip toggle (Bank / Boat / Both) on the precondition block, default Bank. Already required by §8a's `fishing_type` write.

### 3. Venue selection placement → PRECONDITION BLOCK above the wizard

**Decision: keep venue + date + arrival time + fishing_type as a "Session basics" block above the 7-phase wizard.**

`/diary/new` page structure:

```
DiaryNew page
├─ Top: "Session basics" card
│   ├─ Venue (existing inline picker — keep as-is, supports ?venue= querystring)
│   ├─ Date (date input)
│   ├─ Arrival time (time input)
│   └─ Fishing type (Bank / Boat / Both chips)
│   + Big primary CTA: "Build your rig →"
│       (disabled until venue is picked; advances to wizard)
└─ Below (revealed after CTA): SetupWizard with 7 phases
```

**Rationale:**
- The SavedRigsBanner needs `venue.water_type` to filter presets — venue must be resolved before phase 1.
- RN's `SetupScreen.tsx` doesn't pick venue either; venue is chosen at AppRouter level on the previous step. The wizard is rig-only.
- Keeps phase count at 7 (matches RN's named phases exactly: rod / line / leader / style / droppers / flies / spot).

The "Build your rig →" CTA is the explicit transition. Until tapped, the wizard isn't shown — the user is still in basics-capture.

### 4. Weather at commit → LEAVE NULL, async polling handles it

**Decision: take Lovable's recommendation.** Don't fetch weather synchronously in the wizard commit. The existing `poll-session-weather` edge function populates `weather_temp / weather_wind_* / weather_pressure / weather_conditions` on its own schedule.

If the angler manually expanded the optional weather block on the precondition card (existing behaviour, keep it), those manual values do get written. Otherwise null.

## Additional refinement: venue-aware defaults

Lovable's pre-flight noticed extra seasonal-pair columns on `user_profiles`:

```
stillwater_default_rod_weight, stillwater_default_line
river_default_rod_weight,      river_default_line
```

**Use them.** Pre-fill rod weight + line based on `venue.water_type`:

- `venue.water_type === 'stillwater'` (or unset) → `stillwater_default_rod_weight` || `default_rod_weight` || 7
- `venue.water_type === 'river'` → `river_default_rod_weight` || `default_rod_weight` || 5

Same pattern for line: `stillwater_default_line` / `river_default_line` → fall back to `default_line_profile` → `'Floating'`.

**Skip `default_line`** (legacy text column). Use `default_line_profile`.

This makes the wizard's defaults venue-aware for free — no extra UI, no extra schema work.

`{venue.water_type}` resolves to `stillwater` or `river`. If venue is unset (shouldn't happen — wizard is gated on venue) or water_type is null, use the unscoped defaults.
