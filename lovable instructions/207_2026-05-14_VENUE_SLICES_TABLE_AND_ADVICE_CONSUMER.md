# Lovable Prompt 207 — `venue_slices` table + `get-ai-advice-v2` slice consumer

**Date:** 2026-05-14
**Branch / repo:** `Dyninamo/ink-weaver-hub`
**Depends on:** none (pure additive — schema + edge-function change).

**Why this prompt exists:**

The PWA's advice diverges from master/RN advice because the three surfaces draw on different data:

- **Master `scripts/advice_card.py`** — reads per-venue slice JSON (`data/youtube/venue_slices/*.json`) which bakes together `youtube_atoms` (19,370), `reports_enriched` (6,254), `wt_monthly_fly_advice` (1,207), `species_hatch_calendar` (3,841), and per-venue peer-cluster weights. Richest data source.
- **RN `adviceService.ts`** — reads the bundled `FishingDiary/src/data/venue_advice.json` (a stripped 548 KB projection of those same slices). Functionally equivalent to master.
- **PWA `get-ai-advice-v2`** — has access to NONE of the above. It live-ranks `reports_enriched` alone, ignores atoms (Supabase `youtube_atoms` is 0 rows), ignores `wt_monthly_fly_advice` (only the Ghillie chat reads it), ignores hatch calendar, ignores peer-cluster weights. Thinnest data corpus, different code path.

User-visible consequence: ask master for advice on Grafham in May, get one answer. Ask the PWA, get a different one. The user wants them to match.

The fix is to put the slice on Supabase and have `get-ai-advice-v2` read it. The slice contains everything `advice_card.py` needs. 915 slices × ~73 KB avg = ~67 MB of JSONB. Postgres carries it without issue.

This prompt creates the table; the master-side push script `Database/sync/push_venue_slices.py` (already written, will run after deploy) fills it.

**Capture protocol:** per prompt 128, log to `lovable instructions/responses/207_response.md`.

## What this prompt does

1. **§1** — New `venue_slices` table on Supabase. One row per pickable venue, single JSONB column carrying the full slice. Public-read RLS (the slice is already public domain — same logical data already served via the AI prompt).
2. **§2** — `get-ai-advice-v2`: fetch the venue's slice by `venue_id`. When present, use slice.flies_ranked / methods_ranked / top_spots_from_reports / monthly[current_month] / venue_specific_advice as the **primary** tactical grounding inside the LLM prompt. The existing reports_enriched ranking and session_summaries tactical block stay as additive evidence — they're per-user and weather-live, which the slice (baked weekly) is not.
3. **§3** — Telemetry so we can see in `app_events` whether the slice fed each advice request.

No new edge functions. No new client code. No migrations beyond the table create.

## §1 — `venue_slices` table

### §1.1 — Migration

Apply via the Lovable migration tool:

```sql
-- Per-venue baked advice slices, generated by
-- scripts/tactical/build_venue_slice.py on master and pushed via
-- Database/sync/push_venue_slices.py. PWA's get-ai-advice-v2 reads this
-- as the primary tactical grounding so PWA advice matches
-- master/RN advice for the same venue + month.

CREATE TABLE IF NOT EXISTS public.venue_slices (
    venue_id        TEXT PRIMARY KEY,
    slice           JSONB NOT NULL,
    slice_built_at  TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_venue_slices_built_at
    ON public.venue_slices (slice_built_at);

ALTER TABLE public.venue_slices ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Public can read venue_slices"
    ON public.venue_slices FOR SELECT
    USING (true);

CREATE POLICY "Service write venue_slices"
    ON public.venue_slices FOR ALL
    USING (auth.role() = 'service_role')
    WITH CHECK (auth.role() = 'service_role');

-- Touch updated_at automatically on upsert so we can spot stale slices.
CREATE OR REPLACE FUNCTION public.venue_slices_touch_updated()
RETURNS trigger AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_venue_slices_touch_updated ON public.venue_slices;
CREATE TRIGGER trg_venue_slices_touch_updated
  BEFORE UPDATE ON public.venue_slices
  FOR EACH ROW EXECUTE FUNCTION public.venue_slices_touch_updated();
```

The `venue_id TEXT` PK matches `venues_new.venue_id` (also TEXT) — no foreign key constraint because slice builder occasionally produces slices for venues that aren't yet in `venues_new` (during onboarding); we don't want the upsert to fail in that case.

**Foreign key consideration:** if you'd prefer the integrity guarantee, add `FOREIGN KEY (venue_id) REFERENCES venues_new(venue_id) ON DELETE CASCADE` — but accept that the master-side push will reject orphan rows. Recommend **no FK** for now; revisit once the onboarding pipeline is stable.

## §2 — `get-ai-advice-v2` slice consumer

### §2.1 — Fetch slice at the top of the venue-resolved path

Currently, after `venueId` is resolved (around line 540 — where `venues_new` is fetched a second time inside the venue-specific branch), no slice fetch happens.

Add a slice fetch immediately after the `venues_new` fetch in that branch:

```ts
// PWA prompt 207 — fetch the venue's baked slice if present. The slice is
// the canonical advice corpus across master/RN/PWA; when present it should
// drive the tactical block. When absent (new venues, onboarding-in-flight),
// fall through to the existing reports_enriched / session_summaries path
// so we degrade gracefully rather than 404.
const sliceRes = await supabase
  .from("venue_slices")
  .select("slice, slice_built_at")
  .eq("venue_id", venueId)
  .maybeSingle();

const venueSlice: any = sliceRes.data?.slice ?? null;
const sliceBuiltAt: string | null = sliceRes.data?.slice_built_at ?? null;
```

`venueId` is the canonical `venues_new.venue_id` text key — same shape `push_venue_slices.py` uploads under. Use `maybeSingle()` not `single()` so a missing slice returns `null` rather than throwing.

### §2.2 — Project the slice into the LLM prompt

The current prompt builder around line 990–1010 already constructs sections like "Ranked flies:" and "Productive flies:". Add a new "Baked slice (canonical recommendation)" section at the top of the grounding block when a slice is present.

Locate the `prompt = …` template literal that drives the LLM call (the one with `"Ranked flies:"` / `"Best spots:"` / `"Productive flies:"` markers — search for `Ranked flies:` in the file). Insert this above its existing sections:

```ts
const sliceSection = venueSlice ? (() => {
  const monthKey = String(new Date(target_date).getUTCMonth() + 1).padStart(2, "0");
  const monthly = venueSlice.monthly?.[monthKey] ?? {};
  const topFlies = (venueSlice.flies_ranked ?? []).slice(0, 12).map((f: any) =>
    `  - ${f.fly} (score ${f.total_score?.toFixed?.(1) ?? "?"}; ${f.general_rating ?? ""}; videos ${f.distinct_videos ?? 0})`
  ).join("\n");
  const topMethods = (venueSlice.methods_ranked ?? []).slice(0, 8).map((m: any) =>
    `  - ${m.method} (score ${m.yt_score?.toFixed?.(1) ?? "?"})`
  ).join("\n");
  const topSpots = (venueSlice.top_spots_from_reports ?? []).slice(0, 6).map((s: any) =>
    `  - ${s.name} (${s.mentions} mentions)`
  ).join("\n");
  const monthlyFlies = (monthly.top_flies ?? []).slice(0, 6).map((f: any) =>
    typeof f === "string" ? `  - ${f}` : `  - ${f.fly ?? f.name ?? JSON.stringify(f)}`
  ).join("\n");
  const monthlyHatches = (monthly.hatches ?? []).slice(0, 4).map((h: any) =>
    typeof h === "string" ? `  - ${h}` : `  - ${h.name ?? JSON.stringify(h)}`
  ).join("\n");
  const venueSpecific = (venueSlice.venue_specific_advice ?? []).slice(0, 3).map((a: any) =>
    typeof a === "string" ? `  - ${a}` : `  - ${a.text ?? JSON.stringify(a)}`
  ).join("\n");

  return `## Baked slice — canonical recommendation for ${venueSlice.venue?.name ?? "this venue"}
This section is the canonical advice corpus that master and the RN app use.
It blends YouTube atoms (own + peer-cluster + archetype), reports_enriched,
wt_monthly_fly_advice, and species_hatch_calendar with the venue's
peer-cluster weights. Treat it as your primary source; the live-weather
sections below are adjustments on top.

Top flies overall:
${topFlies || "  (none)"}

Top methods overall:
${topMethods || "  (none)"}

Top spots from reports:
${topSpots || "  (none)"}

This month (${monthKey}) — top flies:
${monthlyFlies || "  (none)"}

This month — hatches:
${monthlyHatches || "  (none)"}

Venue-specific advice:
${venueSpecific || "  (none)"}
`;
})() : "";
```

Prepend `sliceSection` to the existing prompt body. The rest of the prompt (live tactical, session-summaries-derived tactical, reports-derived rankings) stays where it is — they're additive context.

### §2.3 — Update the system instruction so the LLM uses the slice as primary

Append `sliceSystemHint` to the existing system prompt string. When the slice is absent, the hint is empty and behaviour is unchanged.

- **Normalised tables** (`venue_slices_flies`, `venue_slices_methods`, etc.) — possible if PostgREST JSONB queries become a bottleneck. Today 73 KB / venue is fine for a single request.

- **Foreign key from `venue_slices.venue_id` → `venues_new.venue_id`** — held until onboarding pipeline stabilises (slices may be built before the venue row lands in `venues_new`).
- **Atoms** — `youtube_atoms` is still 0 rows on Supabase, but the slice already bakes atom-derived rankings, so PWA doesn't need raw atoms to deliver atom-informed advice. Raw atoms can sync separately if a future feature needs them.
